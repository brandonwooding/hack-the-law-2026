# Left-nav, Regimes browser, and editable dossiers

**Date:** 2026-06-27
**Status:** Approved

## Goal

Add a persistent left-side navigation pane to the RegExplorer UI with two
destinations — **Home** (the existing chat workspace) and **Regimes** (a browser
over every top-level regime in the dataset). From the Regimes browser a user can
open a regime's dossier and **manually edit** its prose fields, with edits saved
back to the regime's JSON dossier on disk. Regulatory Guidance is never editable.

## Context / key findings

- A "regime" is an **anchor `Document`** at the `Act` or `Treaty` layer in Neo4j.
  Provisions (`CONTAINS`), cases (`CONSIDERS`), and guidance (`ISSUED_UNDER`)
  hang off it.
- The backend (`db.connect()` via `.env`) targets the **Aura** instance
  (`neo4j+s://0893f8bb…`), which holds the full ~42k-node graph and **5 Act
  anchors**: Online Safety Act 2023, Data Protection Act 2018, Digital Economy
  Act 2017, Investigatory Powers Act 2016, Communications Act 2003.
- The Neo4j **MCP** tool is separately configured against an empty local
  `localhost:7687` DB — it is NOT the app's data source. (See memory note.)
- Editing is already supported end-to-end on the backend: `PUT /regime/{id}`
  (`dossier.save_dossier`) merges changed fields into the JSON dossier and flags
  `edited_by_human`. No backend change is needed for editing.
- `RegimeDetailScreen` already exists but is currently unrouted/unused.
- Routing is TanStack Start file-based; `routeTree.gen.ts` is generated by the
  vite plugin from files under `src/routes/`.

## Decisions

1. **Regime list source:** a new `GET /regimes/all` querying `Act`/`Treaty`
   anchors directly from the graph (returns all 5 today incl. OSA; grows as
   anchors are added). No disk-dossier union needed — the anchors already cover
   the documented set, and a graph query is authoritative.
2. **Editable fields:** prose only — Summary, Scope, Process, Consequence,
   Guidance. Obligations and Regulatory Guidance stay read-only.
3. **Home behaviour:** unchanged setup→workspace flow, wrapped in the new shell.

## Design

### Backend

- `regimes.list_anchor_regimes(driver, database=None) -> list[dict]` — pure-ish
  helper running a Cypher query for `Act`/`Treaty` `Document`s, returning
  `{id, name, short_description}` ordered by citation. `short_description` reuses
  the existing `_short_description` shape (layer/regulator) for consistency.
- `GET /regimes/all` in `api.py` → `{"regimes": [...]}`.
- Test in `tests/test_regimes.py`: feed a fake driver/rows, assert shape and
  ordering. Follows the existing pure-function test pattern (no live Neo4j).

### Frontend

- `src/lib/api.ts`: `fetchAllRegimes(): Promise<RegimeCard[]>` → `GET /regimes/all`.
- **Sidebar shell** (`src/components/research/SidebarLayout.tsx`): persistent
  left nav with brand + links **Home** (`/`) and **Regimes** (`/regimes`),
  active-route highlight, content rendered via `children`. Reuses existing
  design tokens (paper/ink/navy/hairline). Applied in each route's component
  (not the root) so the existing `/` internal header/layout is preserved.
- **Routes:**
  - `/` — unchanged setup→workspace, wrapped in `SidebarLayout`.
  - `/regimes` (`src/routes/regimes/index.tsx`) — fetches `/regimes/all`,
    renders a list of regime cards (name + short description), each linking to
    `/regimes/$id`. Loading / empty / error states.
  - `/regimes/$id` (`src/routes/regimes/$regimeId.tsx`) — renders
    `RegimeDetailScreen` for the id; `onBack` routes to `/regimes`.
- **Edit mode** on `RegimeDetailScreen`:
  - An **Edit** button in the header area toggles edit mode.
  - In edit mode, Summary / Scope / Process / Consequence / Guidance render as
    `<textarea>`s seeded from current values; **Save** and **Cancel** appear.
  - **Save** → `saveRegime(id, {summary, scope, process, consequence, guidance})`
    (`PUT /regime/{id}`), updates local state from the response, exits edit mode.
    Save-in-progress and error states handled inline.
  - **Cancel** discards edits and exits.
  - Obligations and Regulatory Guidance remain read-only in both modes.

## Out of scope (YAGNI)

- Editing obligations (structured list), regime name, or short description.
- Creating/deleting regimes from the Regimes browser.
- Editing the Regulatory Guidance table (explicitly excluded).
- Reworking the Home setup→workspace flow.

## Testing / verification

- Backend: `uv run pytest tests/test_regimes.py` (new test green; suite green).
- Frontend: `npm run build` in `RegExplorerSite` typechecks + builds clean.
- Manual: sidebar navigates Home↔Regimes; a regime opens its dossier; editing a
  prose field and saving persists to `dataset/dossiers/<id>.json` with
  `edited_by_human: true`.
